# Прогон сбора - Руководство администратора

## Обзор

Раздел **"Прогон сбора"** (Collection Runner) — это управляемый pipeline для полного цикла сбора, нормализации и обработки данных маркетплейса. Предназначен для администраторов приложения.

### Доступ

- **Маршрут**: `/admin` → вкладка "Прогон сбора"
- **Требуемая роль**: `admin`, `administrator`, `owner`

---

## Интерфейс "Новый прогон"

### Параметры конфигурации

#### Маркетплейс
- **Wildberries** — основной поддерживаемый маркетплейс
- Yandex Market, Ozon — подготовлены для подключения

#### Режимы сбора

1. **product** — сбор данных по отдельным товарам
   - Введите Article/nmId (по одному на строку)
   - Пример: `123456`, `789012`

2. **seller** — сбор данных по продавцам
   - Введите Seller ID (по одному на строку)
   - Пример: `seller1`, `merchant_12345`

3. **full** — полный сбор товаров и продавцов одновременно
   - Требует и список товаров, и список продавцов

#### Опции обогащения данных

- **Включить отзывы** — добавляет Feedbacks для товаров
- **Включить вопросы** — добавляет Questions для товаров
- **Включить поиск enrichment** — обогащает данные поисковой информацией
- **Расчет unit economics** — автоматически запускает расчет маржинальности и прибыльности

#### Режимы выполнения

- **Dry Run** — использует mock данные вместо реального сбора. **Рекомендуется для тестирования**
  - Отображает источник "Источник данных: mock/fallback"
  - Создает сущности с тестовыми значениями

- **Принудительное обновление** — игнорирует кеш и пересчитывает все значения

#### Параметры производительности

- **Concurrency limit** (по умолчанию 5)
  - Максимальное количество одновременных запросов
  - Увеличивайте с осторожностью во избежание rate limiting

- **Timeout** (по умолчанию 30000 ms)
  - Максимальное время ожидания одного запроса

## Pipeline этапы

Каждый прогон проходит следующие этапы в строгом порядке:

### 1. **validate-input** (Валидация входных данных)
- Проверяет корректность параметров запроса
- Убеждается в наличии обязательных полей
- **Если failed**: прогон останавливается

### 2. **collect-marketplace-data** (Сбор данных)
- Запрашивает данные из маркетплейса (реальные или mock)
- Создает RawMarketplaceFrame для каждого товара/продавца
- **Источник данных**:
  - Real: живой API Wildberries/Yandex/Ozon
  - Mock: генерируются тестовые данные (при `dryRun=true`)
  - Fallback: используется кеш при отсутствии подключения

### 3. **normalize-events** (Нормализация событий)
- Преобразует сырые данные в стандартный формат MarketplaceEvent
- Унифицирует разные источники маркетплейсов

### 4. **save-raw-frames** (Сохранение сырых фреймов)
- Записывает RawMarketplaceFrame в базу данных
- Вычисляет SHA-256 хеш для дедупликации
- Ошибки попадают в DeadLetter с `retryable=true`

### 5. **save-events** (Сохранение нормализованных событий)
- Записывает MarketplaceEvent в базу
- Связывает события с прогоном через `runId`

### 6. **update-snapshots** (Обновление снимков данных)
- Создает или обновляет ProductSnapshot
- Создает или обновляет SellerSnapshot
- Хранит полную копию состояния товара/продавца на момент сбора

### 7. **calculate-unit-economics** (Расчет юнит-экономики)
- **Только если** `includeUnitEconomics=true`
- Запускает расчет маржинальности, contribution margin, точки безубыточности
- Результаты сохраняются в UnitEconomicsSnapshot
- Включает:
  - Валовую прибыль (gross profit)
  - Маржину (margin %)
  - Contribution margin
  - Рентабельность (profitability)

### 8. **verify-results** (Проверка результатов)
- Подсчитывает созданные/обновленные сущности
- Проверяет целостность данных
- Формирует итоговые счетчики

### 9. **build-report** (Построение отчета)
- Создает JSON-отчет прогона
- Включает timeline, счетчики, ошибки
- Доступен для скачивания и анализа

## Текущий прогон - панель мониторинга

### Статусы

- **queued** — ожидает начала обработки
- **running** — в процессе выполнения
- **completed** — успешно завершен
- **failed** — ошибка на одном из этапов
- **cancelled** — отменен пользователем

### Показатели прогресса

- **Progress %** — общий прогресс сбора (0-100%)
- **Этап** — текущий выполняемый этап
- **Счетчики**:
  - Events — всего событий обработано
  - Raw Frames — созданных сырых фреймов
  - Prod Snapshots — обновлено снимков товаров
  - Unit Economics — рассчитано юнит-экономик

### Timeline этапов

Показывает для каждого этапа:
- Иконка статуса (✓ completed, ⚡ running, ⚠ failed, ⏱ pending)
- Имя этапа
- Длительность выполнения (ms)

### Время выполнения

- **Начало**: дата и время запуска
- **Окончание**: дата и время завершения
- **Длительность**: общее время в секундах

## Различие между Real, Mock и Fallback

### Real (реальный сбор)

```
Источник данных: Wildberries API
├─ Живое подключение к маркетплейсу
├─ Требует валидный API ключ
└─ Актуальные данные
```

- Используется при `dryRun=false`
- Требует настройки API ключей в Client
- Может быть rate-limited

### Mock (тестовые данные)

```
Источник данных: Mock generator
├─ Генерирует тестовые данные
├─ Не требует API подключения
└─ Быстро тестирует pipeline
```

- Используется при `dryRun=true`
- **В UI явно указывается**: "Источник данных: mock"
- Идеален для разработки и тестирования
- **Не выдается за реальные данные**

### Fallback (кеш)

```
Источник данных: Fallback/Cache
├─ Используется при отсутствии подключения
└─ Данные могут быть устаревшими
```

- Срабатывает если Real недоступен
- **Отмечается в отчете**

**Важно**: Всегда проверяйте `sourceMode` в отчете перед использованием данных в production!

## Best Practices

### 1. Тестирование конфигурации

```
1. Включите Dry Run
2. Запустите с небольшим набором товаров (1-5)
3. Проверьте отчет и счетчики
4. Отключите Dry Run для реального сбора
```

### 2. Обработка ошибок

```
1. Проверьте таблицу ошибок
2. Нажимайте "Повторить ошибки" для retryable
3. Для невозвратных ошибок — свяжитесь с поддержкой
```

### 3. Мониторинг больших прогонов

```
1. Наблюдайте за прогрессом в режиме реального времени
2. При необходимости нажимайте "Остановить"
3. Пересчитайте ошибки отдельным прогоном
```

### 4. Запуск unit economics

```
1. Убедитесь, что ProductSnapshot заполнены правильно
2. Включите "Расчет unit economics"
3. Проверьте результаты в таблице UnitEconomicsSnapshot
```

## Programmatic control (для разработчиков)

Collection Runner остается admin/demo-инструментом и управляется через `CollectionRunPipelineService`, а production sync выполняется backend scheduler endpoints `/api/admin/scheduled-tasks`:

- `CollectionRunPipelineService.start(request)` — запуск прогона
- `CollectionRunPipelineService.status(runId)` — статус и мониторинг
- `CollectionRunPipelineService.cancel(runId)` — отмена прогона
- `CollectionRunPipelineService.retryErrors(runId)` — повтор ошибок

Подробно см. в разделе "API" админ-панели.

## Troubleshooting

### Прогон зависает на этапе `collect-marketplace-data`

**Решение**: 
- Убедитесь, что API ключ Wildberries активен
- Включите `dryRun=true` для тестирования
- Проверьте timeout в параметрах

### Много ошибок в DeadLetter

**Решение**:
- Проверьте валидность article/nmId
- Убедитесь в доступности маркетплейса
- Попробуйте меньше объектов в одном прогоне

### Unit economics не рассчитаны

**Решение**:
- Убедитесь, что флаг "Расчет unit economics" включен
- Проверьте, что ProductSnapshot заполнены данными о цене
- Дождитесь этапа `calculate-unit-economics` в timeline
